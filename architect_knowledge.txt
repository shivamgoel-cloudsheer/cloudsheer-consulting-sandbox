What is the Salesforce Well-Architected Framework?|||The Salesforce Well-Architected Framework is built around three core principles: Trusted, Easy, and Adaptable. Trusted solutions protect stakeholders with security and compliance, Easy solutions deliver value fast through intentional and automated design, and Adaptable solutions evolve with the business while remaining performant and resilient as demand and scale change.

What are the three pillars of the Salesforce Well-Architected Framework?|||The three pillars are Trusted (protecting the business, users, and data with secure, compliant, and reliable solutions), Easy (delivering value fast through intentional, automated, and engaging design), and Adaptable (ensuring solutions remain available, performant, and resilient as demand, scale, and conditions change). Each pillar contains prescriptive guidance, patterns, and anti-patterns based on knowledge from Salesforce product teams and implementation experts.

What is Organization-Wide Default (OWD) in Salesforce?|||Organization-Wide Defaults (OWD) control the baseline level of access that all users have to each object's records when they do not own the record. OWD settings should be configured to the most restrictive level needed, and then other record-level security tools like role hierarchy, sharing rules, and manual sharing are used to grant additional access to selected users as needed.

How does the Salesforce Role Hierarchy work with OWD?|||The role hierarchy automatically grants record access to users in higher roles for records owned by or shared with users in lower roles. It layers on top of OWD settings and can only grant more access, never less, meaning OWD sets the baseline restriction and the role hierarchy opens up access further for users in higher positions in the hierarchy.

What are the OWD access levels available in Salesforce?|||The OWD access levels include Private (only record owner and users above them in the role hierarchy can access), Public Read Only (all users can view records but only the owner can edit), Public Read/Write (all users can view and edit all records), and Controlled by Parent (access to a child record is determined by the parent record's sharing settings, used for master-detail relationships).

What are Sharing Rules in Salesforce?|||Sharing Rules are automatic exceptions to OWD settings that grant additional record access to specified groups of users. They can be criteria-based (sharing records that match specific field values) or ownership-based (sharing records owned by users in specific roles or groups), and they extend access beyond what OWD and role hierarchy provide without ever restricting it.

What is the Salesforce sharing model architecture?|||The Salesforce sharing model follows a layered approach: OWD sets the most restrictive baseline, the role hierarchy grants upward visibility, sharing rules create automatic exceptions for groups, teams provide record-level collaborative access, and manual sharing allows individual record owners to share specific records. This architecture ensures data security while enabling flexible access patterns across the organization.

What are the key Apex governor limits for synchronous transactions?|||Key synchronous transaction limits include: 100 SOQL queries, 50,000 records retrieved by SOQL, 20 SOSL queries, 150 DML statements, 10,000 records processed by DML, 100 callouts, 10,000 milliseconds CPU time, and 6 MB heap size. These per-transaction limits reset for each Apex transaction and are enforced by the multitenant runtime engine.

What are the key Apex governor limits for asynchronous transactions?|||Asynchronous transactions receive higher limits than synchronous ones: 200 SOQL queries, 50,000 records retrieved by SOQL, 60,000 milliseconds CPU time, and 12 MB heap size. These elevated limits apply to Batch Apex, Queueable Apex, Future methods, and Scheduled Apex, making asynchronous processing essential for data-intensive operations.

What is the daily API request limit in Salesforce?|||Salesforce enforces a rolling 24-hour API request limit calculated as a base amount per edition plus additional requests per user license. Enterprise Edition orgs receive a base of 100,000 daily API requests plus 1,000 additional requests per user license, and requests are tracked on a rolling 24-hour basis rather than a fixed calendar day.

What are the Bulk API limits in Salesforce?|||The Bulk API allows up to 15,000 batch submissions per 24-hour rolling period, shared between Bulk API 1.0 and Bulk API 2.0. Each batch can contain a maximum of 10,000 records with a 10 MB payload limit, and Bulk API operations have a maximum CPU time limit of 60,000 milliseconds on Salesforce servers.

What are the Streaming API limits in Salesforce?|||Streaming API limits include a maximum of 50 PushTopics per org, up to 1,000 subscribers per topic, and a daily limit of 200,000 events. These limits govern the event-driven messaging capabilities used for real-time notifications and data change streaming to external subscribers.

What is considered Large Data Volume (LDV) in Salesforce?|||An object is considered LDV when it contains more than 5 million records and is being actively interacted with, though best practice recommends preparing mitigation strategies when any object crosses 1 million records. LDV management requires careful attention to query optimization, indexing strategies, data archiving, and data skew prevention to maintain acceptable performance.

What is data skew in Salesforce and how do you avoid it?|||Data skew occurs when more than 10,000 child records are associated with the same parent record, causing performance degradation on record locking, sharing recalculation, and DML operations. To avoid data skew, architects should distribute record ownership across multiple parent accounts, avoid assigning excessive child records to a single parent, and design the data model with enough parent records to keep child record counts below the 10,000 threshold.

What are LDV query optimization best practices?|||For LDV query optimization, use indexed fields in SOQL WHERE clauses, avoid NULL checks in queries since indexes cannot be used with nulls, and leverage custom indexes on frequently queried fields. Additionally, use selective queries that return less than 10% of total records (or 333,333 records for standard indexes), and consider skinny tables for frequently accessed large objects to improve query performance.

What data archiving strategies exist for Salesforce LDV?|||Data archiving strategies include using Big Objects for storing historical data natively in Salesforce, leveraging external storage solutions like Heroku Postgres or data warehouses, implementing scheduled batch jobs to move aged records to archive objects, and using third-party archival tools. A best practice is to archive data at the same rate it enters the system to prevent cumulative performance degradation.

What are the five core Salesforce integration patterns?|||The five core integration patterns are: Request and Reply (synchronous real-time calls), Fire and Forget (asynchronous one-way messages), Batch Data Synchronization (scheduled bulk data transfers), Remote Call-In (external system initiates), and Data Virtualization (accessing external data without replication). Each pattern addresses specific timing, volume, and direction requirements for system integration.

What is the Request and Reply integration pattern?|||Request and Reply is a synchronous integration pattern where Salesforce makes a call to an external system and waits for a response before continuing execution. It is suitable for real-time data retrieval and decision-making scenarios where immediate results are required, but architects must consider callout timeout limits (120 seconds for HTTP callouts) and governor limits on the number of callouts per transaction (100 maximum).

What is the Fire and Forget integration pattern?|||Fire and Forget is an asynchronous integration pattern where a source system sends a message without waiting for a response, enabling decoupled communication between systems. This pattern is implemented in Salesforce using Platform Events, outbound messages, or Queueable Apex, and is ideal when the source system does not need immediate confirmation of message processing.

What is the Batch Data Synchronization integration pattern?|||Batch Data Synchronization is used for large-scale periodic data transfers between Salesforce and external systems, typically running on a scheduled basis. This pattern leverages Bulk API 2.0 or Batch Apex to process large datasets in chunks, making it ideal for nightly ETL operations, data warehouse synchronization, and scenarios where real-time updates are not required.

What is the Data Virtualization integration pattern?|||Data Virtualization allows accessing external data in real-time without replicating it into Salesforce, using Salesforce Connect and External Objects to display external data as if it were native Salesforce records. This pattern is optimal when data volumes are too large to store in Salesforce, data changes frequently in the source system, or when regulatory requirements prohibit data duplication.

What is Platform Events in Salesforce?|||Platform Events are custom event messages that apps publish and subscribe to on the Salesforce event bus, enabling event-driven architecture and loosely coupled integration. Unlike Change Data Capture, Platform Events are triggered by custom logic or code and are ideal for building asynchronous workflows, communicating between systems, and publishing domain-specific business events that external subscribers consume.

What is Change Data Capture (CDC) in Salesforce?|||Change Data Capture (CDC) automatically publishes notifications for record creates, updates, deletes, and undeletes to the Salesforce event bus as platform event objects (e.g., AccountChangeEvent for Account). CDC is the preferred mechanism for real-time data synchronization between Salesforce and external systems, requiring no custom code to generate events and supporting both Apex triggers and external CometD subscribers.

What is the difference between Platform Events and Change Data Capture?|||Platform Events are custom-defined event messages triggered by user code or configuration for publishing domain-specific business events, while Change Data Capture automatically captures and publishes data change notifications for standard and custom objects. CDC is best for data synchronization and auditing, while Platform Events are ideal for custom event-driven workflows and cross-system communication that is not tied to specific record changes.

What is Salesforce Shield?|||Salesforce Shield is a suite of three security tools: Shield Platform Encryption (encrypts data at rest with AES-256-bit encryption), Event Monitoring (tracks user activity and API usage with detailed event log files), and Field Audit Trail (retains field history data for up to 10 years beyond the standard 18-month limit). Shield enables organizations to meet compliance requirements for regulated industries.

How does Shield Platform Encryption work?|||Shield Platform Encryption uses AES-256-bit encryption at the field level for standard fields, custom fields, files, and attachments at rest. It relies on a unique tenant secret controlled by the customer combined with a master secret maintained by Salesforce to derive the data encryption key, supporting both probabilistic encryption (most secure, prevents filtering) and deterministic encryption (allows filter-preserving operations on encrypted fields).

What is the difference between Classic Encryption and Shield Platform Encryption?|||Classic Encryption uses a 128-bit AES encryption key and is limited to custom text fields of type Encrypted, while Shield Platform Encryption uses 256-bit AES encryption and can encrypt standard fields, custom fields, files, attachments, and search indexes. Shield also supports Bring Your Own Key (BYOK) and Cache-Only Key management, giving customers full control over their encryption key lifecycle.

What is Event Monitoring in Salesforce Shield?|||Event Monitoring provides visibility into user activity by tracking over 50 event types including logins, API calls, report exports, Apex executions, and Lightning page views through Event Log Files (ELFs). Architects use Event Monitoring data for security incident investigation, adoption analytics, performance monitoring, and compliance auditing, with log data available via API or through the Event Monitoring Analytics app.

What is Salesforce SSO and which protocols does it support?|||Salesforce Single Sign-On (SSO) enables users to access multiple applications with one set of credentials, eliminating repeated logins. Salesforce supports SAML 2.0 (exchanging signed XML assertions between identity and service providers) and OpenID Connect (leveraging OAuth 2.0 flows for identity verification), and can act as both an Identity Provider (IdP) and a Service Provider (SP).

How does SAML-based SSO work with Salesforce?|||In SAML-based SSO, the Identity Provider (IdP) authenticates the user and sends a signed SAML assertion containing user identity and attributes to Salesforce (the Service Provider). Salesforce validates the assertion signature, maps the user identity to a Salesforce user record via the Federation ID or username, and grants access without requiring a separate Salesforce login, supporting both SP-initiated and IdP-initiated flows.

What is Salesforce as an Identity Provider?|||Salesforce can act as a SAML Identity Provider (IdP) or an OpenID Connect Provider, authenticating users and sending identity assertions to external service providers (connected apps). This enables organizations to use Salesforce credentials for SSO into third-party applications, leveraging Salesforce's authentication infrastructure including MFA, session management, and login policies across their application ecosystem.

What does the Identity and Access Management Architect certification cover?|||The Identity and Access Management Architect certification covers identity management concepts, configuring Salesforce to accept third-party identity (SP-initiated SSO), configuring Salesforce as an Identity Provider, access management best practices, and community/portal identity patterns. Candidates must understand SAML, OAuth, OpenID Connect, multi-factor authentication, and session management across Salesforce and external systems.

What is Salesforce Data Cloud (Data 360)?|||Salesforce Data Cloud, renamed Data 360 in October 2025, is Salesforce's customer data platform that unifies customer data from across Sales Cloud, Service Cloud, Marketing Cloud, Commerce Cloud, and external sources into a single customer profile. It operates through a four-stage pipeline of Connect, Harmonize, Analyze, and Activate, using a lakehouse architecture with real-time storage for ingesting, unifying, and activating enterprise data at scale.

What is Zero Copy Architecture in Data Cloud?|||Zero Copy Architecture is a Data Cloud feature that allows organizations to access data where it already lives without physically moving or duplicating it, reducing storage costs and minimizing data movement risks. Through partnerships with Snowflake, Databricks, Google BigQuery, and Amazon Redshift, Zero Copy enables bidirectional data sharing that eliminates traditional ETL processes while maintaining data freshness and governance.

What is Identity Resolution in Data Cloud?|||Identity Resolution is Data Cloud's process of matching and merging customer data from multiple sources into a single unified customer profile. How does Data Cloud ingestion work?|||Data Cloud supports multiple ingestion methods: native Salesforce CRM connector for real-time CRM data, Ingestion API for streaming external data, MuleSoft connectors, and 250+ pre-built connectors for sources like SAP, Shopify, Zendesk, and Workday. What are Data Cloud Segments and Activations?|||Segments are filtered subsets of unified customer profiles in Data Cloud, created using attribute-based criteria, calculated insights, or AI-driven segmentation for targeted marketing and engagement. What is Agentforce?|||Agentforce is Salesforce's autonomous AI agent platform that enables organizations to build, deploy, and manage AI agents capable of handling tasks across sales, service, marketing, and commerce. Powered by the Atlas Reasoning Engine, Agentforce agents operate within configurable Topics that define their scope, use Actions to execute tasks, and follow natural language Instructions with built-in guardrails enforced by the Einstein Trust Layer.

What is the Atlas Reasoning Engine?|||The Atlas Reasoning Engine is the core AI brain powering Agentforce, developed by Salesforce AI Research. It uses a modular architecture comprising planners, action selectors, memory modules, and reflection loops, implementing ReAct-style (Reasoning and Acting) evaluation with System 2 reasoning to break complex goals into steps, select appropriate tools, execute actions, and reflect on outcomes to improve accuracy and reduce hallucinations.

How does the Atlas Reasoning Engine process a request?|||When a user submits a request, Atlas determines intent using an LLM, disambiguates the request into relevant Topics, then enters a reasoning loop: it plans the approach, selects appropriate Actions, executes them, evaluates the results, and reflects on whether the outcome satisfies the original request. Every decision including planning, tool selection, action execution, and reflection is logged for auditing, debugging, and behavior training.

What are Topics in Agentforce?|||Topics are the foundational building blocks of an Agentforce agent, defining what the agent can do and the types of requests it can handle. Each Topic functions as a specialized department with its own scope description, natural language Instructions (business rules and guardrails), and assigned Actions (tools the agent can use), allowing architects to create modular, maintainable agent configurations with clear separation of concerns.

What are Actions in Agentforce?|||Actions are the individual executable tasks that Agentforce agents invoke to accomplish goals, such as querying records, sending emails, creating cases, or calling external APIs. Actions can be built from Flows, Apex classes, prompt templates, or MuleSoft API calls, and are assigned to specific Topics so the Atlas Reasoning Engine can select the appropriate action based on the user's intent and the current conversation context.

What are Instructions in Agentforce?|||Instructions are natural language directives within Topics that guide the agent's behavior, including its persona, conversational tone, business policies, escalation rules, and guardrails on what it can and cannot do. They are provided as context to the LLM during reasoning, and can include both standard prose instructions and structured Agent Script syntax using Logic Instructions (->) for deterministic execution and Prompt Instructions (|) for flexible LLM reasoning.

What is Agentforce Script?|||Agentforce Script is a declarative scripting syntax for defining deterministic agent behaviors within Topics, using two markers: Logic Instructions (->) for actions and business rules that execute deterministically every time, and Prompt Instructions (|) for natural language sent to the reasoning engine for flexible interpretation. This hybrid approach allows architects to combine strict business process compliance with the flexibility of AI-driven conversation handling.

What is the Einstein Trust Layer?|||The Einstein Trust Layer is Salesforce's secure AI architecture that acts as an intermediary between user interactions and Large Language Models, enforcing data privacy, security, and ethical AI use. It includes data masking (removing PII before sending to LLMs), zero data retention (ensuring customer data is never stored by external models), toxic language detection, secure data retrieval with permission enforcement, and a complete audit trail of every prompt through the AI pipeline.

How does the Einstein Trust Layer protect data?|||The Einstein Trust Layer masks personally identifiable information (PII) before prompts are sent to LLMs, encrypts data in flight using TLS, and enforces zero data retention so external models never store customer data. When the response returns, masked data is re-hydrated, output is scanned for toxicity and hallucinations, and the entire journey from prompt creation to response delivery is logged in an audit trail for compliance and accountability.

What is grounding in the Einstein Trust Layer?|||Grounding in the Einstein Trust Layer is the process of enriching AI prompts with relevant, accurate business data from Salesforce to produce contextually accurate responses. Secure data retrieval ensures that grounding data respects the user's sharing and field-level security permissions, so the AI only accesses data the requesting user is authorized to see, preventing unauthorized data exposure through AI-generated responses.

What is MuleSoft API-led connectivity?|||MuleSoft API-led connectivity is a three-layer integration architecture consisting of System APIs (connecting to backend systems of record), Process APIs (orchestrating business logic across multiple System APIs), and Experience APIs (exposing tailored interfaces for specific consumers like mobile apps or web portals). This architecture enables reusable, composable integrations that deliver projects 3-5x faster compared to point-to-point approaches.

What are System APIs in MuleSoft?|||System APIs are the bottom layer of API-led connectivity, providing a clean abstraction over backend systems of record such as Salesforce, SAP, databases, and legacy systems. They function as organizational nouns (Customer, Order, Product, Invoice), hiding backend complexity and data format differences behind a standardized interface, so that changes to underlying systems do not ripple through the entire integration landscape.

What are Process APIs in MuleSoft?|||Process APIs are the middle orchestration layer in API-led connectivity, combining data and functionality from multiple System APIs to execute complete business processes. They represent organizational verbs like ProcessOrder, OnboardCustomer, or CheckInventory, encapsulating business logic and data transformation rules that are independent of both the source systems below and the consuming applications above.

What are Experience APIs in MuleSoft?|||Experience APIs are the top layer of API-led connectivity, exposing tailored data and functionality to specific consumer channels such as mobile applications, web portals, partner systems, or IoT devices. They reshape and filter data from Process APIs to match each consumer's unique requirements, allowing the same underlying business processes and data to serve multiple frontend experiences without duplicating integration logic.

What is Anypoint Platform?|||Anypoint Platform is MuleSoft's unified integration platform for designing, building, deploying, and managing APIs and integrations across cloud and on-premises environments. It includes API Designer (RAML/OAS specification), Anypoint Studio (IDE for building Mule applications), API Manager (governance and policies), Runtime Manager (deployment and monitoring), and Exchange (reusable asset marketplace), providing a complete lifecycle management solution.

What is Heroku Connect?|||Heroku Connect is a data synchronization add-on that replicates data between Salesforce and Heroku Postgres databases bidirectionally. It supports both polling-based synchronization on configurable intervals and near-real-time sync via Salesforce Streaming API, enabling customer-facing applications built on Heroku to seamlessly access and modify Salesforce data without direct API calls to the Salesforce platform.

What are common Heroku Connect use cases?|||Common use cases include building high-traffic B2C web and mobile applications that need Salesforce data without consuming API limits, creating custom analytics platforms that process Salesforce data in Postgres, running field service mobile apps that sync with Salesforce CRM, and consolidating data from multiple Salesforce orgs into a single Heroku Postgres database for unified reporting and cross-org analytics.

What is the single-org vs. multi-org decision in Salesforce?|||The single-org strategy uses one Salesforce instance for the entire enterprise, providing a unified Customer 360 view, simplified administration, and easier cross-departmental reporting. Multi-org uses separate instances for different business units or regions, offering greater customization autonomy, data segregation for compliance, and reduced risk of hitting org-wide limits. The decision depends on data sharing needs, regulatory requirements, organizational complexity, and integration capabilities.

When should you choose a multi-org strategy?|||Choose multi-org when business units have fundamentally different processes with minimal data sharing needs, when regulatory or data residency requirements mandate data segregation by region, when org-level governor limits are being reached, or when acquired companies need to maintain independent operations during transition. Multi-org requires robust integration architecture (typically MuleSoft or Salesforce Connect) to share data across orgs when needed.

What are the challenges of multi-org architecture?|||Multi-org challenges include the need for complex cross-org integration to maintain a unified customer view, duplicated administration and configuration effort across orgs, difficulty implementing global business processes that span multiple orgs, reduced ability to reuse code and configuration, and the need for a master data management strategy to keep shared reference data synchronized. Unified reporting across orgs also requires external solutions like a data warehouse or Tableau CRM.

What are Salesforce deployment strategies?|||Salesforce supports multiple deployment strategies: Change Sets (point-and-click between related orgs), Salesforce CLI/SFDX (command-line metadata deployment with scratch orgs), Unlocked Packages (modular metadata packaging for team-based development), Managed Packages (ISV distribution via AppExchange), and CI/CD pipelines using tools like GitHub Actions, Azure DevOps, or Gearset. Modern best practice favors source-driven development with version control and automated pipelines.

What is Salesforce DX (SFDX)?|||Salesforce DX is Salesforce's modern development framework that introduces source-driven development, scratch orgs (temporary disposable environments), the Salesforce CLI, and a package-based development model. It enables developers to use version control systems like Git as the source of truth for metadata, supports modular development with unlocked packages, and integrates with CI/CD tools for automated testing and deployment pipelines.

What are Scratch Orgs in Salesforce DX?|||Scratch Orgs are temporary, configurable Salesforce environments that are spun up from a scratch org definition file specifying edition, features, and settings. They are fully disposable (lasting up to 30 days), support source tracking to detect metadata changes, and enable developers to work in isolated environments for feature development and testing. Scratch orgs are central to the Salesforce DX development model and CI/CD automation.

What are Unlocked Packages in Salesforce?|||Unlocked Packages are a Salesforce DX packaging format that enables modular, repeatable deployment of metadata components organized by functional area or team. Unlike Managed Packages, unlocked packages allow subscribers to modify the metadata after installation, making them ideal for enterprise teams managing their own org development. They support versioning, dependency management, and can be deployed through CI/CD pipelines for consistent releases.

What is a CI/CD pipeline for Salesforce?|||A Salesforce CI/CD pipeline automates the build, test, and deployment workflow triggered by version control events. It typically includes: source code in Git, automated validation against a scratch or sandbox org, Apex test execution with minimum 75% code coverage, static code analysis (PMD/ESLint), and automated deployment through Salesforce CLI commands. Popular platforms include GitHub Actions, GitLab CI, Azure DevOps, Jenkins, and Salesforce-specific tools like Gearset and Copado.

What is the 75% code coverage requirement in Salesforce?|||Salesforce requires a minimum of 75% overall Apex code coverage across all classes and triggers in an org for production deployment, with no individual class having 0% coverage. This is enforced during deployment validation, and best practice recommends maintaining 80-85% coverage with meaningful assertions rather than just line coverage. Each CI/CD pipeline should run all tests and validate coverage thresholds before deploying to production.

What are Salesforce Change Sets?|||Change Sets are a native Salesforce deployment tool that allows point-and-click metadata transfer between related orgs (connected via deployment connections in the same production hierarchy). They support outbound sets (sending org) and inbound sets (receiving org), are limited to metadata components only (no data), and do not support destructive changes. While simple, they lack version control integration and are being superseded by CLI-based and package-based deployment approaches.

What are Apex design patterns for enterprise development?|||Key Apex design patterns include: Singleton (ensuring one instance per transaction for caching), Service Layer (encapsulating business logic), Selector/Data Access (centralizing SOQL queries), Domain Layer (trigger logic organized by SObject), Unit of Work (managing DML operations and transaction boundaries), and Strategy (interchangeable algorithms). These patterns, often implemented via the fflib/Apex Enterprise Patterns library, promote separation of concerns and testability.

What is the Selector pattern in Apex?|||The Selector pattern centralizes all SOQL queries for a specific SObject into a single class, ensuring consistent field sets, security enforcement (WITH SECURITY_ENFORCED or stripInaccessible), and query optimization across the application. This prevents scattered duplicate queries, makes it easier to add new fields or modify queries in one location, and improves testability by allowing mock selectors in unit tests via dependency injection.

What is the Service Layer pattern in Apex?|||The Service Layer pattern encapsulates business logic into dedicated service classes that act as the API for business operations, separating orchestration logic from triggers, controllers, and batch classes. Service methods accept bulk collections (List or Set of IDs), enforce sharing rules via 'with sharing', and coordinate calls to Selector, Domain, and Unit of Work classes, providing a clean boundary that can be exposed via REST endpoints or invocable actions.

What is the Domain Layer pattern in Apex?|||The Domain Layer pattern organizes trigger logic into SObject-specific domain classes that encapsulate validation, defaulting, and business rules for each object. Built on the fflib_SObjectDomain base class, domain classes handle trigger events (before insert, after update, etc.) and keep trigger handlers clean and testable. This pattern prevents trigger logic from becoming monolithic and enables consistent behavior whether records are modified via UI, API, or Apex code.

What is the Unit of Work pattern in Apex?|||The Unit of Work pattern manages DML operations by registering new, dirty, and deleted records in a single transaction scope, then committing all changes at once in the correct dependency order. Using fflib_SObjectUnitOfWork, it handles parent-child relationship resolution automatically, reduces scattered DML statements (helping stay within the 150 DML governor limit), and supports transaction rollback for error handling.

What is the Singleton pattern in Apex?|||The Singleton pattern ensures only one instance of a class exists per transaction, typically used for caching query results, configuration settings, or custom metadata to avoid redundant SOQL queries within the same execution context. It is implemented using a private static instance variable, a private constructor, and a public static getInstance() method, helping conserve governor limits by reusing expensive query results across multiple method calls.

What is Salesforce Shield Field Audit Trail?|||Field Audit Trail extends standard field history tracking from 18 months to up to 10 years, storing historical field values in the FieldHistoryArchive big object. It supports up to 60 fields per object (versus the standard 20-field limit for field history tracking) and enables compliance with regulatory requirements like GDPR, HIPAA, and financial regulations that mandate long-term data retention for audit purposes.

What are the API types available in Salesforce?|||Salesforce offers multiple API types: REST API (lightweight JSON/XML for CRUD and queries), SOAP API (enterprise XML-based with strong typing), Bulk API 2.0 (asynchronous processing of large datasets up to millions of records), Streaming API (push notifications via CometD/Bayeux protocol), Metadata API (deploy and retrieve org configuration), Tooling API (developer tooling and metadata), and Pub/Sub API (gRPC-based event streaming for Platform Events and CDC).

What is the Pub/Sub API in Salesforce?|||The Pub/Sub API is a gRPC-based API that provides efficient, bidirectional streaming for publishing and subscribing to Platform Events and Change Data Capture events. It replaces the legacy CometD-based Streaming API with better performance, supports binary payload encoding with Avro schemas, provides flow control for backpressure management, and enables high-throughput event processing for real-time integration scenarios.

What is Salesforce Connect (External Objects)?|||Salesforce Connect enables real-time access to external data without copying it into Salesforce, using External Objects that map to data stored in external systems via OData 2.0/4.0, custom Apex adapters, or cross-org adapters. External Objects support SOQL queries, relationships with standard/custom objects, list views, and record detail pages, making them ideal for data virtualization when dealing with large external datasets or compliance restrictions on data replication.

What are Custom Metadata Types in Salesforce?|||Custom Metadata Types are a configuration framework for storing application metadata that is deployable, packageable, and queryable without counting against SOQL governor limits in certain contexts. They are ideal for storing configuration settings, mapping tables, business rules, and feature flags that need to move between environments via change sets or packages, unlike Custom Settings which are considered data and require separate migration.

What are Custom Settings vs. Custom Metadata Types?|||Custom Settings store org-specific or user-specific data values accessed via an in-memory cache (no SOQL needed for hierarchy settings), but they are treated as data and require manual migration between environments. Custom Metadata Types store configuration metadata that deploys with packages and change sets, supports SOQL queries, and is better suited for application configuration that must be consistent across environments. Custom Metadata Types are the preferred modern approach.

What is the Salesforce Governor Limit for SOQL queries?|||The SOQL query limit is 100 queries per synchronous transaction and 200 per asynchronous transaction. Each query can retrieve a maximum of 50,000 records, and aggregate queries count toward the same limit. To optimize, architects use the Selector pattern to centralize queries, leverage relationship queries to reduce query count, and use SOQL for loops for processing large result sets without exceeding heap limits.

What is the Salesforce Governor Limit for DML statements?|||The DML statement limit is 150 per transaction, with a maximum of 10,000 records processed across all DML statements combined. Each DML call (insert, update, delete, undelete, merge) counts as one statement regardless of the number of records in the collection. The Unit of Work pattern helps consolidate DML operations, and bulkification ensures that collections are processed in single DML calls rather than individual record operations.

What is the Salesforce Governor Limit for callouts?|||The callout limit is 100 HTTP or web service callouts per synchronous transaction, with a maximum timeout of 120 seconds per individual callout and a cumulative timeout of 120 seconds for all callouts. Long-running integrations should use asynchronous patterns (Queueable Apex with callout=true, or continuation objects in Lightning) to avoid blocking the user transaction and to leverage the higher asynchronous limits.

What is the CPU time limit in Salesforce?|||The CPU time limit is 10,000 milliseconds (10 seconds) for synchronous transactions and 60,000 milliseconds (60 seconds) for asynchronous transactions. CPU time measures actual server processing time excluding callout wait time, DML wait time, and queued time. Exceeding this limit throws a LimitException that cannot be caught, making it critical to optimize loops, avoid unnecessary recursion, and use efficient data structures.

What is the heap size limit in Salesforce?|||The heap size limit is 6 MB for synchronous transactions and 12 MB for asynchronous transactions. Heap size measures the total memory used by all variables, objects, and collections in the execution context. To manage heap size, architects use SOQL for loops (which process records in chunks of 200), avoid storing large collections in memory, and leverage transient variables in Visualforce controllers to exclude data from view state.

What are Salesforce testing best practices for architects?|||Testing best practices include: writing meaningful assertions (not just achieving coverage), testing bulk scenarios with 200+ records to validate bulkification, using Test.startTest()/Test.stopTest() to reset governor limits, implementing the TestDataFactory pattern for consistent test data creation, mocking external callouts with HttpCalloutMock and WebServiceMock, testing both positive and negative scenarios, and validating sharing and security enforcement in tests running as different user profiles.

What is the TestDataFactory pattern?|||The TestDataFactory pattern centralizes test data creation into a utility class with static methods that generate SObject records with all required fields populated, supporting optional parameter overrides. This eliminates duplicate test data setup across test classes, ensures consistency, and makes tests resilient to validation rule changes. Factory methods should create records in bulk (e.g., createAccounts(Integer count)) and support both inserted and non-inserted record generation.

How do you test governor limits in Apex?|||Use Test.startTest() and Test.stopTest() to establish a separate governor limit context for the code being tested, isolating it from setup DML and queries. The code between these calls receives fresh governor limits, allowing you to test that your implementation stays within limits when processing realistic data volumes. Additionally, use Limits class methods (Limits.getQueries(), Limits.getDMLStatements()) to assert that operations consume an expected number of resources.

What is performance optimization in Salesforce?|||Performance optimization encompasses query tuning (selective SOQL with indexed fields), bulkification (processing collections instead of single records), efficient use of caching (Custom Settings, Platform Cache, Static variables), minimizing view state in Visualforce, lazy loading in Lightning components, leveraging asynchronous processing for heavy operations, and designing data models to avoid data skew. Architects should also use the Query Plan tool and Debug Logs to identify performance bottlenecks.

What is Platform Cache in Salesforce?|||Platform Cache provides a temporary in-memory data store (using key-value pairs) divided into Org Cache (shared across all users and requests) and Session Cache (scoped to individual user sessions). It reduces repetitive SOQL queries and callouts by caching frequently accessed data, supports configurable TTL (time-to-live), and provides up to 10 MB for Enterprise Edition (expandable with add-on licenses). Architects use it for caching configuration data, API tokens, and computed values.

What is the Salesforce Metadata API?|||The Metadata API enables programmatic deployment and retrieval of org configuration metadata including custom objects, fields, page layouts, Apex classes, Flows, and profiles. It is the underlying API used by Salesforce CLI (sf project deploy/retrieve), Change Sets, and CI/CD tools for managing metadata across environments. The Metadata API supports both file-based and CRUD-based operations, with the deploy() call accepting a ZIP file of metadata components.

What is the Tooling API in Salesforce?|||The Tooling API provides REST and SOAP interfaces for building developer tools, supporting operations like running Apex tests, querying metadata component details, creating trace flags for debug logging, and accessing code coverage results. Unlike the Metadata API which handles full deployments, the Tooling API is optimized for fine-grained developer operations and powers tools like the Developer Console, VS Code extensions, and custom development utilities.

What are permission sets and permission set groups?|||Permission Sets grant additional permissions to users beyond their profile, following a least-privilege model where profiles provide minimum access and permission sets layer on additional capabilities. Permission Set Groups bundle multiple permission sets into a single assignable unit, simplifying administration. With the Salesforce shift away from profile-based permissions, the recommended approach is to use a minimal base profile and assign all functional permissions through permission sets and permission set groups.

What is Field-Level Security (FLS) in Salesforce?|||Field-Level Security controls which fields users can see and edit on each object, enforced through profiles and permission sets. FLS is not automatically enforced in Apex code, so architects must use WITH SECURITY_ENFORCED in SOQL, Security.stripInaccessible() for DML operations, or Schema.SObjectType.describe() checks to ensure code respects FLS. Lightning Data Service and standard controllers enforce FLS automatically.

What is the difference between with sharing and without sharing in Apex?|||The 'with sharing' keyword enforces the running user's sharing rules and record-level access in an Apex class, while 'without sharing' bypasses sharing enforcement and grants access to all records. 'inherited sharing' defers to the calling class's sharing context. Best practice for architects is to default to 'with sharing' for security, use 'without sharing' only when system-level access is explicitly required (e.g., utility classes), and document the security rationale.

What are Named Credentials in Salesforce?|||Named Credentials are a declarative framework for configuring authenticated callout endpoints, storing the URL, authentication protocol, and credentials in a single referenceable configuration. They support OAuth 2.0, JWT Bearer, Password Authentication, AWS Signature, and custom authentication protocols, and are referenced in Apex callouts or External Services to avoid hardcoding endpoints and credentials. Named Credentials are deployable via metadata and are the recommended approach for all external callouts.

What are External Credentials in Salesforce?|||External Credentials (introduced with the Named Credentials redesign) separate authentication configuration from endpoint URL configuration, allowing multiple Named Credentials to share the same authentication setup. They define the authentication protocol and credential mapping, while Named Credentials define the endpoint URL and reference the External Credential. This separation enables per-principal authentication where different users can authenticate with different credentials to the same external service.

What is Salesforce Flow architecture?|||Salesforce Flow is the declarative automation platform encompassing Screen Flows (user-facing guided processes), Record-Triggered Flows (replacing Process Builder and Workflow Rules), Scheduled Flows (time-based automation), and Platform Event-Triggered Flows. Flows execute within the same governor limit context as Apex triggers, support transaction control with pause elements, and can call Apex actions via invocable methods, making them the primary low-code automation tool recommended by Salesforce.

What are the governor limit considerations for Flows?|||Flows share the same per-transaction governor limits as Apex, including 100 SOQL queries, 150 DML statements, and 10,000 CPU milliseconds for synchronous execution. Each Get Records element consumes one SOQL query, each Create/Update/Delete Records element consumes one DML statement, and loops with DML inside can quickly exhaust limits. Best practices include bulkifying Flows with collection variables, minimizing elements inside loops, and using before-save Record-Triggered Flows for field updates to avoid DML operations.

What is a before-save vs. after-save Record-Triggered Flow?|||Before-save flows run before the record is committed to the database, can modify field values on the triggering record without consuming a DML statement (making them highly efficient), but cannot create/update related records or perform callouts. After-save flows run after the record is committed, can perform DML on other objects, make callouts, and send notifications, but consume additional DML statements and SOQL queries. Architects should use before-save flows for field defaulting and validation, and after-save flows for cross-object operations.

What is the order of execution in Salesforce?|||The Salesforce order of execution for record saves is: system validation, before-save flows, before triggers, system validation and duplicate rules, after triggers, assignment rules, auto-response rules, workflow rules, after-save flows, entitlement rules, rollup summary calculations, criteria-based sharing evaluation, and finally commit. Understanding this order is critical for architects to predict behavior when multiple automations interact on the same object.

What are the different types of Salesforce sandboxes?|||Salesforce offers four sandbox types: Developer (5 MB data, metadata only, refresh every 1 day), Developer Pro (1 GB data, metadata only, refresh every 1 day), Partial Copy (5 GB data, metadata plus sampled data via template, refresh every 5 days), and Full Copy (same storage as production, complete data copy, refresh every 29 days). Architects design environment strategies using these sandbox types for development, QA, UAT, staging, and integration testing.

What is Salesforce Big Objects?|||Big Objects are a Salesforce storage solution designed for massive data volumes (billions of records) that would exceed standard object limits. They support two types: Standard Big Objects (like FieldHistoryArchive) and Custom Big Objects (user-defined). Big Objects use a composite primary key for indexing, support only Async SOQL and Salesforce Connect queries, and are ideal for archiving historical data, audit trails, and IoT telemetry while keeping the core org performant.

What is Async SOQL in Salesforce?|||Async SOQL is a query method designed for querying Big Objects and large standard objects, running queries asynchronously in the background and writing results to a target object. Unlike standard SOQL which has a 50,000 record retrieval limit, Async SOQL can process millions of records, making it suitable for analytical queries over archived data in Big Objects and for data transformation operations on very large datasets.

What is Salesforce Connect vs. Heroku Connect?|||Salesforce Connect (External Objects) provides real-time read access to external data through OData or custom Apex adapters without data replication, ideal for viewing external data in Salesforce UI. Heroku Connect physically replicates data between Salesforce and Heroku Postgres bidirectionally, ideal for building custom applications that need fast database access to Salesforce data. Choose Salesforce Connect for virtualization and Heroku Connect for application development requiring local data access.

What is the Composite API in Salesforce?|||The Composite API allows executing multiple REST API requests in a single HTTP call, supporting up to 25 subrequests in a single composite request and up to 5 subrequests in a composite graph. Subrequests can reference the results of previous subrequests, enabling dependent operations like creating a parent Account and child Contacts in one call. This reduces round trips and API call consumption, making it valuable for complex integration scenarios.

What is the difference between REST API and SOAP API in Salesforce?|||REST API uses lightweight JSON or XML over HTTP with standard verbs (GET, POST, PATCH, DELETE), is stateless, and is preferred for modern integrations, mobile apps, and web applications. SOAP API uses XML with WSDL contracts, provides strong typing and formal service descriptions, and is better suited for enterprise middleware that requires strict contracts. Both share the same daily API call limits, but REST API is generally recommended for new integrations due to its simplicity and performance.

What are the Salesforce multi-org integration options?|||Multi-org integration options include: Salesforce Connect Cross-Org Adapter (real-time data virtualization between orgs), Salesforce-to-Salesforce (native record sharing with publish/subscribe model), MuleSoft (enterprise middleware for complex orchestration), Pub/Sub API (event-driven cross-org communication), direct REST/SOAP API calls between orgs, and Heroku as a hub (consolidating data from multiple orgs in Heroku Postgres). The choice depends on data volume, latency requirements, and integration complexity.

What is Salesforce-to-Salesforce?|||Salesforce-to-Salesforce enables native record sharing between separate Salesforce orgs using a publish/subscribe model where organizations establish connections and share specific objects and fields. Records are shared as read-only or read-write, changes auto-sync between connected orgs, and it supports standard and custom objects. While simple to configure, it has limitations on data volume and complexity compared to middleware solutions like MuleSoft.

What are the key considerations for Salesforce performance at scale?|||Key performance considerations include: selective SOQL queries using indexed fields, avoiding data skew by distributing ownership, implementing skinny tables for frequently queried large objects, using Platform Cache to reduce redundant queries, leveraging asynchronous processing for bulk operations, optimizing page load times with lazy loading and cacheable wire methods, implementing pagination for large list views, and regular monitoring via Event Monitoring and the Salesforce Optimizer tool.

What are skinny tables in Salesforce?|||Skinny tables are custom database tables maintained by Salesforce Support that contain a subset of frequently accessed fields from a standard or custom object, excluding soft-deleted records and archived data. They provide faster query performance for LDV objects by reducing I/O overhead, but must be requested through Salesforce Support, are not visible in SOQL, and are automatically applied by the query optimizer when they can improve performance.

What is the Query Plan tool in Salesforce?|||The Query Plan tool (accessible in Developer Console under Help > Use Tooling API) analyzes SOQL query execution plans, showing the cost estimate, selectivity, and which indexes will be used. A query with a cost below 1.0 is considered selective and will use an index; above 1.0 triggers a full table scan. Architects use this tool to optimize LDV queries, validate that custom indexes are being utilized, and troubleshoot slow-performing queries.

What is Apex Managed Sharing?|||Apex Managed Sharing (also called programmatic sharing) allows Apex code to create sharing records in the object's share table (e.g., AccountShare, CustomObject__Share) to grant record-level access beyond OWD, role hierarchy, and sharing rules. Share records specify the UserOrGroupId, AccessLevel (Read or Edit), and RowCause (manual or custom). This provides the most granular sharing control and is used when criteria-based sharing rules cannot address complex access requirements.

What is an Apex trigger framework?|||An Apex trigger framework is an architectural pattern that delegates trigger logic from the trigger file to handler classes, promoting separation of concerns, testability, and maintainability. Common frameworks include the TriggerHandler pattern (virtual class with before/after methods), Kevin O'Hara's framework, and the fflib Domain Layer. Best practice is one trigger per object delegating to a handler, with recursion control, bulkification, and clear separation between validation, field defaulting, and cross-object logic.

What is the role of a Salesforce Technical Architect in a project?|||A Salesforce Technical Architect leads solution design across the full project lifecycle, from stakeholder discovery and requirements analysis through solution design, build oversight, data migration planning, deployment strategy, and adoption. They evaluate trade-offs between declarative and programmatic solutions, ensure scalability and governance compliance, manage technical risks, lead development teams, and communicate architecture decisions clearly to both technical and business stakeholders.

What should an architect consider when designing for regulatory compliance?|||Architects must evaluate data residency requirements (which may mandate multi-org or specific data center regions), implement Shield Platform Encryption for data at rest, configure Field Audit Trail for long-term audit requirements, enforce FLS and sharing model for data access control, enable Event Monitoring for security audit trails, implement data retention and deletion policies for GDPR right-to-be-forgotten, and design consent management flows for privacy regulations.

What is the Salesforce event bus?|||The Salesforce event bus is the underlying pub/sub messaging infrastructure that transports Platform Events, Change Data Capture events, and other event messages between producers and consumers. Events published to the bus are stored for up to 72 hours (configurable retention), enabling replay capabilities for missed events. Consumers can be Apex triggers, Flows, Lightning components via empAPI, or external systems via the Pub/Sub API or legacy CometD clients.

What is event replay in Salesforce?|||Event replay allows subscribers to retrieve events published up to 72 hours in the past by providing a replay ID that indicates the position in the event stream to resume from. Subscribers can use ReplayId -1 (new events only), -2 (all retained events), or a specific ReplayId to resume from a known position. This capability ensures reliable event delivery even when subscribers experience temporary outages, preventing data loss in event-driven architectures.

What is the Pub/Sub API vs. CometD for event consumption?|||The Pub/Sub API is a modern gRPC-based streaming API that provides bidirectional event streaming with flow control, Avro-encoded binary payloads, and support for both publishing and subscribing, replacing the legacy CometD-based Streaming API. CometD uses HTTP long-polling with JSON payloads and only supports subscribing. The Pub/Sub API offers better performance, lower latency, and is the recommended approach for new event-driven integrations.

How do you handle governor limit errors in production?|||Implement proactive monitoring using Limits class methods to track consumption during execution, set up email alerts for approaching API and storage limits, use Event Monitoring to identify limit-hitting transactions, and design circuit breaker patterns in integration code. For reactive handling, use try-catch blocks (noting that LimitException for CPU and heap cannot be caught), implement retry logic with exponential backoff for callout failures, and use Platform Events to decouple processing into separate transaction contexts.

What is the maximum number of records in a Salesforce org?|||There is no hard maximum on total records in a Salesforce org, as storage is determined by the edition and number of user licenses. Enterprise Edition provides 10 GB of data storage plus 20 MB per user license for data, and 10 GB plus 20 MB per user for file storage. Additional storage can be purchased, and architects should implement data lifecycle management including archiving to Big Objects and external systems to manage growth proactively.

What is the transaction control mechanism in Salesforce?|||Salesforce supports Database.setSavepoint() and Database.rollback() for explicit transaction control within Apex, allowing partial rollback to a savepoint while preserving earlier DML. Each trigger or automation block executes within the same transaction context, and a DML exception in any part rolls back the entire transaction. Platform Events with the Publish After Commit behavior respect transaction boundaries, while Publish Immediately events persist even if the transaction rolls back.

What are the best practices for Apex exception handling?|||Best practices include: creating custom exception classes extending the Exception base class, using try-catch-finally blocks for recoverable errors, logging exceptions to a custom Error Log object or Platform Events for monitoring, never silently swallowing exceptions, using Database.insert(records, false) for partial success processing, implementing retry patterns for transient callout failures, and centralizing error handling in the Service Layer to provide consistent error responses across all entry points.

What is the difference between Queueable and Future methods?|||Queueable Apex supports complex parameter types (SObjects, custom Apex types), allows job chaining (calling another Queueable from execute()), provides a job ID for monitoring, and supports transaction finalizers for error handling. Future methods (@future) only accept primitive parameters, cannot be chained, and offer no job tracking. Queueable Apex is the recommended modern approach for asynchronous processing, though both share the same asynchronous governor limits.

What is Batch Apex and when should you use it?|||Batch Apex processes large datasets by dividing records into batches of up to 2,000 records (default 200), executing each batch in a separate transaction with fresh governor limits. Use Batch Apex when processing more records than can fit in a single transaction (beyond 50,000 SOQL records or 10,000 DML records), for scheduled data cleanup or transformation jobs, and for operations that require callouts to external systems on a per-record basis (with Database.AllowsCallouts interface).

What are Transaction Finalizers in Salesforce?|||Transaction Finalizers (System.Finalizer interface) attach to Queueable Apex jobs and execute after the Queueable completes, regardless of success or failure. They receive a FinalizerContext with the job result (SUCCESS or UNHANDLED_EXCEPTION), exception details, and the async job ID. Finalizers enable automatic retry logic for failed jobs (up to 5 retries), error logging, and cleanup operations, providing robust error handling for asynchronous processing chains.

What is the Lightning Component framework architecture?|||Lightning components use a component-based architecture with two models: Aura Components (legacy, using the Aura framework with client-side controllers) and Lightning Web Components (LWC, using modern web standards with JavaScript, HTML templates, and Shadow DOM). LWC is the recommended standard, offering better performance through direct browser rendering, reactive properties with @wire for data binding to Apex or Lightning Data Service, and support for component composition and events.

What is Lightning Data Service (LDS)?|||Lightning Data Service provides a client-side caching layer and data access framework that loads, creates, updates, and deletes records without requiring Apex code, automatically enforcing CRUD, FLS, and sharing rules. LDS uses a shared record cache across all components on a page, eliminating redundant server calls and ensuring data consistency. It is accessed via wire adapters like getRecord, getFieldValue, and createRecord in Lightning Web Components.

What is the difference between Aura and Lightning Web Components?|||Aura Components use a proprietary framework with a component bundle structure (component, controller, helper, renderer), a custom event model, and client-side rendering that adds overhead. Lightning Web Components (LWC) use standard JavaScript classes, HTML templates, and CSS with Shadow DOM encapsulation, leveraging native browser APIs for superior performance. LWC supports modern JavaScript features, has a smaller footprint, and is Salesforce's recommended component model for all new development.

What are the Agentforce licensing considerations for architects?|||Agentforce operates on a consumption-based pricing model charging per conversation rather than per user license, with each conversation defined as a completed interaction between an agent and a customer or employee. Architects must plan for conversation volume estimation, implement guardrails to prevent runaway agent interactions, design Topics and Actions to maximize first-contact resolution, and configure appropriate escalation paths to human agents when the AI agent cannot resolve the request.

What is the MuleSoft Composer vs. Anypoint Platform decision?|||MuleSoft Composer is a low-code integration tool designed for business users and admins to build simple point-to-point integrations using pre-built connectors and a visual flow builder, without coding. Anypoint Platform is the full enterprise integration platform for developers and architects, supporting API-led connectivity, custom transformations, complex orchestration, and governance at scale. Architects choose Composer for simple, admin-managed integrations and Anypoint Platform for enterprise-grade integration architecture.

What is the recommended approach for Salesforce data migration?|||Data migration best practices include: mapping source fields to target Salesforce objects with transformation rules, cleansing data before migration (deduplication, standardization), loading in the correct order to maintain relationship integrity (parent objects before children), using External ID fields for upsert operations to handle re-runs, leveraging Bulk API 2.0 for large volumes, implementing validation rules and duplicate management after migration, and planning for iterative trial loads with rollback capabilities.

What are External IDs and their role in data migration?|||External ID fields are custom fields marked as External ID that store unique identifiers from external systems, enabling upsert operations that insert new records or update existing ones based on the external identifier. They are automatically indexed for query performance, support relationship mapping during data loads (allowing child records to reference parent External IDs instead of Salesforce IDs), and are essential for ongoing integration and data migration scenarios.

What is the Salesforce Optimizer?|||The Salesforce Optimizer is a free tool that analyzes your org and generates a report with actionable recommendations across areas including storage usage, custom field utilization, inactive users, unused automation, code coverage gaps, and feature adoption. Architects use it for org health assessments, identifying technical debt, planning cleanup initiatives, and as part of regular governance reviews to maintain org performance and reduce complexity over time.

How should architects approach Salesforce technical debt?|||Architects should conduct regular org health assessments using Salesforce Optimizer and code analysis tools, maintain a technical debt backlog prioritized by business impact and risk, establish governance processes that review new customizations before deployment, refactor duplicate automation (especially migrating legacy Workflow Rules and Process Builders to Flows), clean up unused fields, objects, and code, and allocate a percentage of each release cycle to debt reduction work.

What is the recommended approach for Salesforce org governance?|||Org governance should include a Center of Excellence (CoE) with defined roles and decision-making authority, documented standards for naming conventions, coding standards, and automation guidelines, a change management process with architecture review gates, environment strategy with clear promotion paths (Dev > QA > UAT > Prod), regular org health reviews, a release management calendar, and training programs to maintain the skill level of administrators and developers.

What are the Agentforce agentic design patterns?|||Key agentic patterns include: Single Agent (one agent handling one domain), Multi-Agent Collaboration (cooperative agent swarms with specialized agents for sub-tasks), Human-in-the-Loop (agent escalation to human agents with context transfer), Tool Augmentation (agents using Actions to call APIs, query data, or trigger Flows), and Retrieval Augmented Generation (RAG, grounding agent responses in knowledge base articles and CRM data via the Atlas Reasoning Engine).

What is the difference between deterministic and probabilistic AI in Agentforce?|||Deterministic behaviors in Agentforce are defined using Agent Script Logic Instructions (->) that execute the same way every time, ensuring compliance with strict business rules like refund policies or escalation protocols. Probabilistic behaviors use the LLM-based Atlas Reasoning Engine for flexible natural language understanding and dynamic decision-making. Architects should use deterministic paths for regulated processes and probabilistic reasoning for open-ended customer interactions.

What are Data Cloud Calculated Insights?|||Calculated Insights are custom metrics and aggregations computed on unified Data Cloud profiles using SQL-like expressions, enabling advanced analytics without moving data to external systems. They can aggregate transaction history, compute customer lifetime value, calculate engagement scores, or derive propensity models, and the results are available for segmentation, activation, and consumption by Agentforce agents and Einstein AI features for real-time decisioning.

What is the Data Cloud Data Model (DMO)?|||The Data Cloud Data Model uses Data Model Objects (DMOs) organized around standardized entity categories: Profile (individuals and accounts), Engagement (interactions and events), Other (supporting entities), and Data Lake Objects (raw ingested data). DMOs follow a canonical schema that enables identity resolution and cross-source data harmonization, with mapping rules transforming ingested source data into the standardized DMO structure during the Harmonize phase.

What is the recommended multi-cloud architecture for Salesforce?|||A multi-cloud architecture typically centers on Data Cloud as the unification layer connecting Sales Cloud, Service Cloud, Marketing Cloud, Commerce Cloud, and external systems. MuleSoft provides the integration backbone using API-led connectivity, Agentforce delivers AI-powered automation across clouds, and the Einstein Trust Layer secures all AI interactions. Architects should establish shared identity resolution in Data Cloud, consistent event-driven patterns via Platform Events, and unified governance through the Well-Architected Framework.
